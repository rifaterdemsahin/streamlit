📅 Creation Order — Step by Step

Each step below represents the order in which files and infrastructure should be created, together with the rationale explaining why that order matters.

1
Step 1 · Concept & Context
📋 Define the Objective & Conventions
📄 copilot.md 📄 real.md 📄 formula_architecture.md 📄 environment.md

Before writing a single line of code, document the OKRs, tech-stack decisions, and coding conventions. These files feed AI pair-programming tools (GitHub Copilot) and ensure every contributor shares the same mental model. environment.md is written first because it clarifies which secrets and URLs must exist before deployment can succeed.

⬇
2
Step 2 · Python Dependencies
📦 Pin Runtime Libraries
📄 requirements.txt

Declaring dependencies before writing application code ensures reproducibility. Streamlit ≥ 1.32, pandas, NumPy, Plotly, and yfinance are pinned here. Both the local pip install workflow and the Docker build reference this file, so it is the single source of truth for the Python environment.

⬇
3
Step 3 · Analysis Logic
🔬 Pure Helper Functions
🐍 utils.py

utils.py contains only pure functions with no Streamlit imports: compute_returns(), compute_rolling_average(), compute_z_scores(), get_numeric_columns(), get_sample_data(), and try_fetch_yfinance(). Separating these from the UI layer means they can be unit-tested independently and reused in scripts or notebooks.

⬇
4
Step 4 · Web Application
🖥️ Streamlit UI Entry Point
🐍 app.py

The main application imports from utils.py and builds four analysis tabs: 📋 Data · 📊 Stats · 📉 Charts · 💡 Insights. Written after utils.py because the UI is purely a consumer of the helper layer — this keeps the dependency arrow pointing in only one direction.

⬇
5
Step 5 · Containerisation
🐳 Docker Image Definition
🐳 Dockerfile

A multi-stage build: Stage 1 (builder) installs dependencies into /opt/venv; Stage 2 (runtime) copies only the venv and app source into a slim Python 3.11 image, running as a non-root appuser. The Dockerfile is written after the app code is stable because it packages exactly those two files (app.py + utils.py) and requirements.txt.

⬇
6
Step 6 · Infrastructure Config
☁️ Fly.io Configuration Template
⚙️ fly.toml.example ⚙️ fly.toml generated

fly.toml.example is committed as a reference configuration showing the expected shape (port 8501, health-check path, auto-stop settings). The live fly.toml is generated by fly-setup.sh at make create time and is gitignored when customised, preventing app-name collisions between contributors.

⬇
7
Step 7 · Developer Automation
🔧 Scripts & Makefile
🔧 scripts/check-prereqs.sh 🔧 scripts/fly-setup.sh 🔧 scripts/fly-destroy.sh 📋 Makefile

Shell scripts use set -euo pipefail and coloured ok / info / warn / die helpers for a polished developer experience. The Makefile wraps all scripts with memorable short commands (make create, make deploy, make destroy) and declares check-prereqs as a dependency of every lifecycle target to prevent half-initialised states.

⬇
8
Step 8 · CI/CD & GitHub Pages
🤖 Automated Pipelines
⚙️ .github/workflows/deploy.yml ⚙️ .github/workflows/static.yml 🌐 index.html 📄 README.md 📄 contributor.md

deploy.yml triggers on every push to main and runs flyctl deploy --remote-only — no local Docker daemon required. static.yml deploys index.html to GitHub Pages, substituting the __FLY_APP_URL__ placeholder with the FLY_APP_URL secret so the iframe always points at the correct Fly.io backend. Documentation (README.md) and contributor.md are finalised last when all behaviour is confirmed.

💡 Rationale Behind Each Decision

🐍
Why Streamlit?
Streamlit turns Python data-science scripts into interactive web apps with zero front-end code. A single streamlit run app.py gives us tabs, sliders, charts, and data-upload — ideal for a PoC that must demonstrate value quickly without a JavaScript framework.
📊
Why pandas + Plotly + yfinance?
pandas handles tabular data manipulation and filtering. Plotly renders interactive, publication-quality charts (heatmaps, time-series, histograms). yfinance provides free, real-time market data for AAPL, MSFT, GOOGL — making the PoC immediately useful without any API keys.
🐳
Why multi-stage Docker?
Stage 1 (builder) installs build-essential and compiles any C-extension wheels. Stage 2 (runtime) copies only /opt/venv and the two app files, keeping the final image small and free of build toolchain. A non-root appuser reduces the attack surface.
☁️
Why Fly.io?
Fly.io supports fly deploy --remote-only — no local Docker daemon needed. Auto-stop / auto-start scales the VM to zero when idle (cost ≈ $0) and wakes it in ~2 s on the next request. The free tier fits a PoC with < 100 daily users on a $1/month budget.
⚙️
Why a Makefile?
Short, memorable commands (make create) abstract away multi-step flyctl sequences. check-prereqs is declared as a dependency of every lifecycle target so you can never accidentally run make deploy without a validated environment.
🤖
Why GitHub Actions CI/CD?
Every push to main triggers deploy.yml which calls flyctl deploy --remote-only with only the FLY_API_TOKEN secret. No credentials are ever hardcoded. Failed deployments surface immediately as a red check in the PR status.
🌐
Why GitHub Pages + iframe?
GitHub Pages gives a stable, branded URL (rifaterdemsahin.github.io/streamlit/) for free. The index.html iframe embeds the Fly.io backend, so the compute runs on Fly.io while the entry-point URL never changes. The __FLY_APP_URL__ placeholder is replaced at deploy time, so updating the backend URL requires only a secret change — not a code edit.
📄
Why utils.py separation?
Pure helper functions live in utils.py with no Streamlit imports. This means they can be unit-tested without starting a Streamlit server and can be imported by future scripts, notebooks, or data pipelines without dragging in the entire UI stack.
